A Fourth Agent, an HTTP Front Door, and One Command to See It All Work

Three change requests, three shipped pieces: a configurable policy rule engine slotted into the agent chain, a REST API gateway that puts the file-based pipeline behind ordinary HTTP calls, and demo.sh - a script that starts the system, submits every sample transaction, and prints the results with nothing left for a human to click.

The Brief

Three requests, read verbatim

The update request specified exactly three additions to the existing three-agent pipeline. Each is mapped below to what actually shipped.

Request 1agents/rule_engine.py

"A new agent + configurable rule engine"

  • Rules live in rules_config.json, not in Pythonfield / operator / value / action - edit JSON, not code
  • Sits between fraud scoring and compliancereject terminates; flag adds to compliance's evidence
Request 2api/server.py

"A REST API gateway... so transactions can be submitted and results retrieved via API calls"

  • FastAPI over the existing pipelineno business logic duplicated - calls integrator.py directly
  • 7 routes: submit, list, get, summary, health, plus rulesGET/PUT /rules reconfigures the policy engine live, no file edit
Request 3demo.sh

"A demo script... starts everything, submits test transactions, and displays results with zero manual steps"

  • One command, full round tripvenv -> server -> 8 submissions -> summary -> teardown
  • No jq, no extra dependencystdlib urllib for HTTP, python -m json.tool to render

Architecture - Now Four Agents

The rule engine slots in, not on top

Same file-based protocol, one more hop. A configurable policy check now sits between risk scoring and final disposition.

AGENT 1

transaction_validator

Required fields, positive Decimal amount, valid ISO 4217 currency. Unchanged.

AGENT 2

fraud_detector

0-1 risk score from value, timing, cross-border, and instrument signals. Unchanged.

AGENT 3

rule_engine

Evaluates rules_config.json: sanctioned-country -> reject; large withdrawal or branch wire -> flag.

AGENT 4

compliance_checker

Final call - now also reads policy_flags alongside fraud risk and cross-border checks.

HTTP Front Door

api/server.py, driven end to end by demo.sh

The gateway is a thin adapter - every route calls straight into integrator.py or rule_engine.py. demo.sh is the client that proves it.

Routes
POST/transactionsrun one transaction through all four agents
GET/transactionslist every stored result
GET/transactions/{id}fetch one - 404 if unknown
GET/summarycounts by final status, run-level
GET/healthreadiness probe for demo.sh
GET/rulesread the policy engine's current config
PUT/rulesreplace it - validated, live on the next submission
./demo.sh, in order
  1. Reset shared/ for a clean runsame effect as integrator.py --clear
  2. Start uvicorn, wait on /healthpolling loop, not a fixed sleep
  3. POST all 8 sample transactionsprints id + final status as each returns
  4. GET /summary, pretty-print, tear downtrap kills the server on exit, even on error
